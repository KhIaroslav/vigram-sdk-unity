    Source Analysis

    Summary

    The current source material is a Unity Package located at Packages/com.vigram.sdk-gh1. It is not a full Unity sample project with scenes, prefabs, or serialized setup. The reliable sources of truth are package.json, README.md, CHANGELOG.md, assembly definition files, Runtime/Editor C# sources, and the Android/iOS native plugin folders.

    SDK version 1.0.0 targets mobile Unity applications on iOS and Android. It wraps Bluetooth LE transport, viDoc peripheral protocol handling, GNSS/NMEA models, NTRIP correction streams, laser measurements, device motion, file logging, and SDK token authentication.

    Confirmed From Source

    • Minimum Unity version: 2021.3.
    • Package name: com.vigram.sdk-gh1.
    • Target platforms: iOS, Android.
    • Runtime assemblies: ViGram and ServicesAssembly.
    • ViGram.asmdef references the precompiled VigramAuthenticityVerificator.dll.
    • The Android plugin includes BLE permissions and BluetoothLE.androidlib.
    • The iOS post-build processor adds NSBluetoothAlwaysUsageDescription.
    • The public 1.0.0 API uses PascalCase members and interface names with the I prefix.
    • NTRIP in 1.0.0 has explicit success/failure callbacks for connection startup and an error channel for correction data.
    • IGPSService is created with both IPeripheralService and IBluetoothService.

    Public Documentation Boundary

    Public documentation should focus on the API that application developers can use safely:

    • Vigram factories.
    • IAuthenticationService and AuthenticationResult.
    • IBluetoothService, BPeripheral, and CBManagerState.
    • IPeripheralService callbacks and high-level request methods.
    • NmeaMessage, DeviceMessage, SatelliteMessage, and related public value types.
    • INtripService and INtripTaskService.
    • IGPSService.
    • ILaserService and LaserConfiguration.
    • IDeviceService.
    • ILogger.

    The documentation should not expose test-only or internal implementation details as integration guidance:

    • APIs compiled only under VIGRAM_TEST_API.
    • Internal Bluetooth connection result types.
    • Authentication payload/cache/backend internals.
    • Parser segments, transport buffers, protocol framing, and raw parser implementation classes.
    • Low-level Send(byte[], int) as a normal first-run path. It can be mentioned only as an advanced escape hatch.
    • Generated or internal data structures that are not intended to be used directly by app code.

    PDF Material Assessment

    The legacy SDK_Unity.pdf is useful as a scenario inventory. It lists the flows that users expect to find in SDK documentation: authentication, Bluetooth scan, peripheral startup, NMEA callbacks, device version and battery requests, GNSS configuration, NTRIP, GPS, laser measurements, and logging.

    The PDF should not be used as a copy-paste source for code. It contains pre-1.0.0 naming such as Vigram.Init, Vigram.bluetoothService, startScan, connect, and peripheral.nmea(...). It also shows older signatures for GPS and NAV-DOP/NAV-PVT callbacks. The modern docs should preserve the use cases, but rewrite the examples against the current public API.

    Assumptions

    • SDK tokens are issued by an external viGram process. The package source does not include a public token issuance flow.
    • Production apps should depend on an immutable version tag such as 1.0.0.
    • Device selection UI is application-specific. SDK examples should show how to receive BPeripheral values, not prescribe a UI framework.
    • Some runtime behavior, such as platform permission prompts, depends on Unity project settings and target OS versions.
